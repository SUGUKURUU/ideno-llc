# スマホ対応していないサイトが失うもの｜モバイルファーストの重要性を解説

「うちのサイトはパソコンで見ればきれいなんだけど、スマホで見るとちょっと崩れるんですよね」——このような話を、中小企業の経営者や担当者から頻繁に聞きます。「ちょっと崩れる」という感覚は、実際にはビジネス上の大きな損失を生んでいます。

結論から言えば、**スマートフォン非対応のサイトは、訪問者の半数以上を失い、検索順位も下がり、問い合わせも来なくなります**。本記事では、モバイル非対応が引き起こす具体的な損失を数値で示しながら、モバイルファースト設計の基本と改善方法を解説します。

---

## 結論：スマホ非対応サイトが失う3つのもの

### 1. 訪問者の半数以上

総務省の通信利用動向調査によると、インターネット利用デバイスとしてスマートフォンを使う人の割合は年々増加し続けており、多くの業種でWebサイトへのアクセスの50〜70%がスマートフォンからとなっています。スマートフォンで閲覧しにくいサイトは、この大多数の訪問者をすぐに離脱させます。

### 2. 検索順位

Googleは2019年から「モバイルファーストインデックス」を正式に採用しています。これは、Googleのクローラーがサイトを評価する際に、**PCではなくスマートフォン版のコンテンツを基準にする**という方針です。スマートフォン対応が不十分なサイトは、SEO評価が下がり、検索結果での表示順位が低下します。

### 3. 問い合わせ・購入の機会

モバイルで使いにくいフォームや読みにくいテキストは、問い合わせへの心理的摩擦を高め、直前でのコンバージョン離脱を招きます。せっかく広告費や制作費をかけて集客しても、スマートフォン体験が悪いと投資対効果が著しく低下します。

## 背景・現状：「スマホ対応」の現在地

![スマホ非対応で失う3つの損失](/blog/images/mobile-first-website/01-mobile-neglect-losses.svg)

### モバイルファーストインデックスとは

Googleはかつて、PCサイトのコンテンツを主な評価対象としていました。しかし、モバイルユーザーが多数派になったことを受け、評価の基準をモバイルサイトに切り替えました。これがモバイルファーストインデックス（MFI）です。

MFIの導入以降、スマートフォン対応が不十分なサイトは以下のようなペナルティを受けます。
- 検索ランキングの低下
- クロールの優先度低下（インデックスされにくくなる）
- 検索結果でのリッチスニペット（スキーマ）の表示機会が減少

### Core Web Vitalsとモバイル

Googleが2021年に導入した「Core Web Vitals（コアウェブバイタル）」は、ページ体験の品質を評価する3つの指標です。

| 指標 | 内容 | 目標値 |
|---|---|---|
| LCP（Largest Contentful Paint） | メインコンテンツの表示速度 | 2.5秒以内 |
| FID（First Input Delay）→INP | ユーザー入力への反応速度 | 200ms以内 |
| CLS（Cumulative Layout Shift） | ページの視覚的安定性（コンテンツの突然のずれ） | 0.1以下 |

これらの指標はモバイル環境（3G/4Gのモバイル回線、スマートフォン端末）を基準に測定されます。PCでは問題ないスコアでも、モバイルでは基準を下回るケースが多く、中小企業のサイトではCLSとLCPで課題を抱えているケースが特に多い傾向があります。

### 日本の中小企業のモバイル対応の現状

Web制作・運用の現場では、以下のような「モバイル対応の落とし穴」がよく見られます。

- 数年前に制作したサイトがレスポンシブ非対応のまま
- 「一応スマホでも表示される」が文字サイズが小さすぎて読みにくい
- ボタンが小さくてタップしにくい
- 横スクロールが発生する
- フォームがスマートフォンで入力しにくい
- 画像が未圧縮で表示速度が遅い

特に2015年以前に制作されたサイトは、レスポンシブデザイン（画面サイズに応じて自動的にレイアウトが変わる設計）が採用されていないケースが多く、今でも「PCでは問題ないがスマホでは使いにくい」状態のまま運用されています。

## 実践方法・解決策：モバイルファーストを実現する7つのステップ

![モバイル対応チェックリスト](/blog/images/mobile-first-website/02-mobile-optimization-checklist.svg)

### ステップ1：現状のモバイル対応状況を診断する

まず自社サイトの現状を把握しましょう。以下の無料ツールで診断できます。

**Google Search Console（無料）**
- 「モバイルユーザビリティ」レポートで、モバイル表示の問題箇所を一覧確認できます
- エラーが検出された場合、具体的な修正方法も案内されます

**Google PageSpeed Insights（無料）**
- URLを入力するだけでモバイルとPCの表示速度スコアを確認できます
- Core Web Vitalsの各指標も確認できます
- スコア90以上：良好、50〜89：要改善、49以下：問題あり

**Google モバイルフレンドリーテスト（無料）**
- サイトがモバイルフレンドリーかどうかを即座に判定します
- 現在はPageSpeed Insightsに統合されつつあります

### ステップ2：レスポンシブデザインを採用する

レスポンシブデザインとは、一つのHTMLで画面サイズに応じてレイアウトを自動調整する設計手法です。現在のWeb標準として定着しており、新規サイト制作では必須条件です。

**レスポンシブ対応のポイント**
- CSSのメディアクエリで各画面サイズに対応したスタイルを定義する
- 「モバイルファースト」の考え方でスマートフォンのデザインから先に設計し、PCに拡張していく
- フレックスボックスやCSSグリッドを活用して柔軟なレイアウトを実現する

**旧サイトのリニューアル判断基準**
- スマートフォンで横スクロールが発生する → リニューアル推奨
- 文字サイズが12px以下のコンテンツがある → 修正必要
- 制作から5年以上経過している → レスポンシブ非対応の可能性が高い

### ステップ3：フォントサイズとタップターゲットを最適化する

スマートフォンでの読みやすさの基準は以下の通りです。

**フォントサイズ**
- 本文テキスト：16px以上（Googleの推奨）
- 見出し（H2）：20〜24px程度
- 注釈・補足テキスト：14px以上（12px以下は避ける）

**タップターゲットのサイズ**
- ボタン・リンクのタップ領域：最低44×44px（Apple Human Interface Guidelinesの推奨）
- ボタン間の余白：8px以上（誤タップを防ぐ）
- フォームの入力フィールド：高さ44px以上

### ステップ4：画像を最適化して表示速度を改善する

画像の最適化は表示速度改善で最も効果が出やすい施策です。

**画像最適化のチェックリスト**
- [ ] WebP形式を採用する（JPEGと比較して25〜34%のサイズ削減）
- [ ] 表示サイズに合った画像を用意する（2,000px幅の画像を200pxで表示しない）
- [ ] 圧縮ツール（TinyPNG、Squoosh など）で画像ファイルサイズを削減する
- [ ] `loading="lazy"` 属性で画面外の画像の遅延読み込みを設定する
- [ ] ファーストビューの画像には `fetchpriority="high"` を設定する

**目安となるファイルサイズ**
- ヒーロー画像：200KB以内
- サービス説明の画像：100KB以内
- アイコン・ロゴ：SVG形式を推奨

### ステップ5：モバイル向けナビゲーションを設計する

PCのナビゲーションメニューをそのままスマートフォンに転用すると、ほぼ確実に使いにくくなります。

**モバイルナビゲーションの基本**
- ハンバーガーメニュー（三本線アイコン）を採用し、タップで開閉する
- メニュー項目を3〜7項目程度に絞る
- 問い合わせ・電話などのCTAはヘッダーに固定表示する
- スクロール量が多いページでは「ページトップへ戻る」ボタンを設置する

**固定フッターCTAの活用**
スマートフォンで最も効果的なCTA配置の一つが、画面下部に固定表示するフッターCTAです。「電話で問い合わせ（電話アイコン）」「メールで問い合わせ（メールアイコン）」をスティッキーに表示することで、どのページを閲覧していても問い合わせに即アクセスできます。

### ステップ6：フォームをモバイルフレンドリーに最適化する

問い合わせフォームはコンバージョンの最終ステップです。スマートフォンでの入力体験を徹底的に改善しましょう。

**フォーム最適化のポイント**
- `type="email"` を設定してメール専用キーボードを呼び出す
- `type="tel"` を設定して数字キーボードを呼び出す
- `autocomplete` 属性を設定してブラウザの自動入力を活用できるようにする
- フォームフィールドの間隔を十分に取り、誤タップを防ぐ
- ラジオボタン・チェックボックスのタップ領域を大きくする

### ステップ7：継続的なモバイルUX改善サイクルを構築する

モバイル対応は「一度やれば完了」ではなく、継続的な改善が必要です。

**月次で確認すべき指標**
- Google Search ConsoleのCore Web Vitals（モバイル）スコア
- Googleアナリティクスのモバイルデバイスのセッション数・直帰率・CVR
- PageSpeed InsightsのLCP・CLS・INP

**半年〜1年ごとに見直すべき項目**
- 主要ページをスマートフォン実機で手動チェック
- 新しいスマートフォン機種・画面サイズへの対応確認
- 競合他社のモバイルUXとの比較

## よくある失敗

### 失敗1：「PCで見てきれいだから大丈夫」という誤解

Web担当者やデザイナーがPCで作業・確認するため、「PCで見ると問題ない」というバイアスが生まれやすい環境があります。必ずスマートフォン実機（複数機種）での表示確認を行い、実際の操作感をチェックすることが不可欠です。

### 失敗2：レスポンシブ対応だが「コンテンツの最適化」がされていない

「レスポンシブデザインだから大丈夫」と安心しているケースがありますが、レスポンシブはあくまでレイアウトの自動調整であり、コンテンツの最適化とは別物です。PCでは適切な画像サイズでもスマートフォンでは大きすぎる、テキストが多すぎてスクロールが長い、などの問題は別途対応が必要です。

### 失敗3：表示速度の改善を後回しにする

「デザインや機能を先に完成させてから速度対応する」という進め方は、後になるほど改善コストが増加します。制作段階から画像最適化・不要スクリプトの排除・CDNの設定を組み込んでおくことが重要です。

### 失敗4：「スマホ対応」と「スマホ最適化」を混同する

スマートフォンで「表示される」ことと「使いやすい」ことは別物です。「一応表示される」レベルのサイトは、Googleの評価もユーザー体験も中途半端になります。表示されるだけでなく、スマートフォンユーザーが快適に使えるレベルまで最適化することが目標です。

### 失敗5：古いWordPressテーマをそのまま使い続ける

WordPress を使っているサイトで、数年前に購入した古いテーマを更新せずに使い続けているケースがあります。古いテーマはモバイル対応が不十分なものが多く、Core Web Vitalsのスコアも低い傾向があります。テーマの更新、または現代的なモバイル対応テーマへの切り替えを検討してください。

## 成功事例（一般化・実名不使用）

### 事例A：飲食店チェーンのスマホ対応リニューアル

地域で複数店舗を展開する飲食チェーンが、スマートフォン対応のリニューアルを実施した事例。

**リニューアル前の状況**
- スマートフォンからのアクセス：全体の65%
- スマートフォンの直帰率：82%（PCは51%）
- スマートフォンからの予約完了率：PCの40%程度

**実施した改善内容**
- レスポンシブデザインへの全面移行
- 予約フォームをスマートフォン向けに再設計（入力項目を8→4項目に削減）
- 固定フッターに「電話予約」「オンライン予約」ボタンを設置
- 画像をWebP形式に変換し、PageSpeedスコアを45→82に改善

**リニューアル後の変化（3ヶ月後）**
- スマートフォンの直帰率：82%→58%に改善
- スマートフォンからの予約完了率：PCとほぼ同等水準まで回復
- Google検索での表示回数：前年同期比+35%

### 事例B：BtoBメーカーのCore Web Vitals改善

製造業のBtoBメーカーが、Google Search ConsoleでCore Web Vitalsの警告を受けて改善に取り組んだ事例。

**警告の内容**
- LCP（Largest Contentful Paint）：4.8秒（目標2.5秒以内）
- CLS（Cumulative Layout Shift）：0.35（目標0.1以下）

**改善施策**
- ヒーロー画像を2.4MBから180KBに圧縮（WebP変換）
- ファーストビューの大型バナーに`fetchpriority="high"`を設定
- 広告スクリプトの非同期読み込みによるCLS改善
- フォント読み込みの最適化

**改善後の結果**
- LCP：4.8秒→1.9秒（目標値クリア）
- CLS：0.35→0.07（目標値クリア）
- Google検索の表示順位が平均2〜3ポジション改善

## まとめ

スマートフォン非対応のサイトが失うものは「訪問者・検索順位・問い合わせ」の3つです。モバイルファーストは「あれば良い機能」ではなく、現代のWebビジネスにおける必須条件です。

**モバイルファースト対応のポイント総まとめ**
- Googleモバイルファーストインデックスにより、スマホ対応がSEOの前提条件
- Core Web VitalsはモバイルでのLCP・CLS・INPを改善することが重要
- レスポンシブデザインの採用はスマホ対応の出発点
- フォントサイズ・タップターゲット・フォームもスマホ向けに最適化する
- 画像の圧縮・WebP変換・遅延読み込みで表示速度を改善する
- Google Search Console・PageSpeed Insightsで定期的にモニタリングする

現状のサイトのモバイル対応状況に不安がある方は、まず無料ツールで診断するところから始めてみてください。

## FAQ

**Q1. スマホ対応のリニューアルにかかる費用はどのくらいですか？**

A. 既存サイトのレスポンシブ対応のみであれば10〜30万円程度が目安です。WordPressサイトのテーマ変更であれば5〜15万円程度の場合もあります。ただし、コンテンツの見直し・フォームの再設計・表示速度改善まで含めると30〜80万円程度になることが多いです。完全リニューアルの場合は50万円以上が相場です。

**Q2. スマホ対応を自社で対応することはできますか？**

A. WordPressを使っている場合、モバイルフレンドリーなテーマへの切り替えであれば自社対応も可能です。ただし、テーマ変更によってデザインが崩れるリスクや、既存コンテンツの移行作業が発生するため、バックアップを取ってから慎重に進める必要があります。コードに自信がない場合は制作会社に依頼することをおすすめします。

**Q3. AMPは採用すべきですか？**

A. AMP（Accelerated Mobile Pages）はGoogleが推進していたモバイル高速化技術ですが、現在はGoogleのAMP優遇が縮小されており、必須ではなくなっています。Core Web Vitalsの改善（画像最適化・表示速度改善）に注力することの方が現実的かつ効果的です。

**Q4. モバイル専用サイト（m.example.com）とレスポンシブ、どちらがよいですか？**

A. 現在はレスポンシブデザインを強く推奨します。モバイル専用サイトは管理が二重になり、コンテンツの更新ミスが起きやすく、SEOの評価も分散するリスクがあります。Googleも公式にレスポンシブデザインを推奨しています。

**Q5. タブレット対応も必要ですか？**

A. タブレット（iPad等）への対応も重要ですが、優先度は スマートフォン > タブレット > PCの順です。多くのレスポンシブデザインはタブレット向けのブレークポイントも設定されているため、スマートフォン対応を適切に行えばタブレットも概ね問題なく表示されます。ただし、実機確認は行うようにしてください。

**Q6. 古いAndroid端末や古いiOSのサポートはどこまで必要ですか？**

A. 一般的にはiOS 14以降、Android 8以降のサポートを目安とするケースが多いです。Google Analyticsでサイトへのアクセス端末を確認し、古いOSのシェアが1%未満であれば対応コストに見合わない可能性があります。制作会社と相談の上、サポート範囲を事前に合意しておくことをおすすめします。

**Q7. スマホ対応改善後、SEO効果が出るまでどのくらいかかりますか？**

A. Core Web VitalsやモバイルUXの改善は、Googleのクロール・インデックス更新サイクルの関係で、効果が検索順位に反映されるまで数週間〜3ヶ月程度かかるのが一般的です。Google Search ConsoleでCore Web Vitalsのスコアが「良好」に改善されたことを確認してから、検索順位の変化をモニタリングしてください。
